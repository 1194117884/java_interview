# ✅服务端渲染和客户端渲染的区别

# 典型回答

**客户端渲染（CSR - Client Side Rendering）**：浏览器下载HTML骨架和JavaScript文件后，由JavaScript在客户端执行渲染，生成最终的页面内容。React默认就是CSR模式。

**服务端渲染（SSR - Server Side Rendering）**：服务器在接收到请求后，在服务端执行React组件渲染生成完整的HTML，然后将HTML直接发送给浏览器。浏览器不需要再执行大量JS就能展示内容。

```jsx
// CSR流程
浏览器请求 → 获取空白HTML + JS bundle → 下载JS → 执行React渲染 → 显示页面
// 用户先看到白屏，然后页面瞬间出现

// SSR流程
浏览器请求 → 服务器执行React渲染 → 返回完整HTML → 浏览器直接显示
// 用户立即看到内容（但还不能交互）
```

**核心区别**：
- **首屏加载**：SSR直接返回完整HTML，首屏内容立即可见；CSR需要等待JS下载执行
- **SEO**：SSR搜索引擎可以直接抓取到完整内容；CSR搜索引擎可能无法抓取JS生成的内容
- **交互时间**：CSR的HTML和JS一起下载，HTML渲染后可立即交互；SSR需要HTML下载后额外下载JS进行水合（hydration）才能交互
- **服务器负载**：SSR需要服务器承担渲染计算；CSR服务器压力小

# 扩展知识

### 水合（Hydration）过程

SSR的核心是水合，它将服务端生成的静态HTML与客户端的React事件和状态绑定：

```jsx
// 服务端：将React组件渲染为HTML字符串
import { renderToString } from 'react-dom/server';

const html = renderToString(<App />);
// 输出: <div data-reactroot=""><h1>Hello</h1><button>Click</button></div>

// 客户端：接管服务端生成的DOM，绑定事件处理
import { hydrateRoot } from 'react-dom/client';

hydrateRoot(document.getElementById('root'), <App />);
// React会复用已有的DOM节点，只绑定事件和状态
// 不会重新创建DOM
```

```
水合时间线：
    服务器                          浏览器
    │                               │
    ├─ renderToString(<App>)        │
    │   → 生成HTML                  │
    ├─ 发送HTML ──────────────────→ ├─ 显示内容（可见但不可交互）
    │                               │
    │                               ├─ 下载JS bundle
    │                               ├─ 执行hydrateRoot()
    │                               │   → 绑定事件处理器
    │                               ├─ 页面可交互
    │                               │
```

### CSR vs SSR vs SSG 对比

| 维度 | CSR（客户端渲染） | SSR（服务端渲染） | SSG（静态生成） |
|------|----------------|-----------------|---------------|
| 渲染位置 | 浏览器 | 服务器 | 构建时 |
| 首屏速度 | 慢（先白屏） | 快（直接展示） | 最快（静态HTML） |
| SEO | 差（爬虫难解析） | 好（完整HTML） | 最好 |
| TTI（可交互时间） | 与FCP接近 | 晚于FCP | 与FCP接近 |
| 服务器负载 | 低 | 高（每次请求都渲染） | 无（纯静态文件） |
| 动态数据 | 实时获取 | 实时获取 | 构建时固定 |
| 部署复杂度 | 低 | 高（需要Node服务器） | 低 |
| 适用场景 | 后台管理、内部系统 | 内容站、电商 | 博客、文档站 |

### CSR的详细流程

```bash
CSR时序图：
Time 0ms:      浏览器请求页面
Time 100ms:    服务器返回近乎空的HTML + <script src="bundle.js">
Time 200ms:    开始下载bundle.js（可能几百KB）
Time 1500ms:   bundle.js下载完成
Time 1500ms+:  React解析JSX、执行渲染、创建DOM
Time 2000ms:   用户看到页面（首次内容渲染 - FCP）
Time 2500ms+:  所有图片和数据加载完成（LCP）

交互时间 ≈ FCP（因为JS已经加载，事件绑定即时生效）
```

### SSR的详细流程

```bash
SSR时序图：
Time 0ms:      浏览器请求页面
Time 30ms:     服务器开始执行renderToString（取决于组件复杂度）
Time 200ms:    服务器返回完整HTML文档（包含内容）
Time 200ms:    浏览器开始解析HTML，直接显示内容（FCP极快）
Time 200ms:    浏览器开始下载JS bundle（可能几百KB）
Time 1500ms:   bundle.js下载完成
Time 1500ms+:  React开始hydrate（复用已有DOM，绑定事件）
Time 2000ms+:  hydrate完成，页面可交互（TTI）

交互时间 ≈ 下载JS时间 + hydrate时间
```

### SSR的主要挑战

```jsx
// 挑战1：水合不匹配（Hydration Mismatch）
// 服务端和客户端渲染结果不一致

function TimeDisplay() {
  // ❌ 时间在服务端和客户端不一致
  return <div>{new Date().toLocaleString()}</div>;
  // 服务端渲染："2024-01-01 12:00:00"
  // 客户端水合："2024-01-01 12:00:05"
  // 水合不匹配！
}

// ✅ 使用useEffect解决
function TimeDisplay() {
  const [time, setTime] = useState(null);
  useEffect(() => {
    setTime(new Date().toLocaleString());
  }, []);
  // SSR时显示null或loading状态，客户端再更新
  return <div>{time || '加载中...'}</div>;
}
```

```jsx
// 挑战2：服务端无法访问浏览器API
function WindowSize() {
  // ❌ 服务端渲染时，window不存在
  const [width, setWidth] = useState(window.innerWidth);
  return <div>{width}</div>;
}

// ✅ 使用useEffect或动态导入
function WindowSize() {
  const [width, setWidth] = useState(1024); // 默认值
  useEffect(() => {
    setWidth(window.innerWidth);
    const handle = () => setWidth(window.innerWidth);
    window.addEventListener('resize', handle);
    return () => window.removeEventListener('resize', handle);
  }, []);
  return <div>{width}</div>;
}
```

### Next.js 的数据获取模式

```jsx
// Next.js 提供了多种数据获取策略

// SSR（每次请求都渲染）
export async function getServerSideProps(context) {
  const data = await fetch('https://api.example.com/data');
  return { props: { data } };
}

// SSG（构建时生成一次）
export async function getStaticProps() {
  const data = await fetch('https://api.example.com/data');
  return { props: { data }, revalidate: 3600 }; // ISR增量静态生成
}

// ISR（增量静态生成）：在SSG基础上，定期重新生成
// revalidate: 3600 表示每小时重新生成一次

// CSR（客户端获取）
function ClientComponent() {
  const [data, setData] = useState(null);
  useEffect(() => {
    fetch('https://api.example.com/data').then(r => r.json()).then(setData);
  }, []);
  return <div>{data?.name}</div>;
}
```

### 现代框架的SSR解决方案

```bash
传统SSR方案：
- Next.js（React全栈框架）
- Remix（Web标准优先的React框架）
- Razzle（自定义SSR配置）

核心优化技术：

1. 流式渲染（Streaming SSR）
   - React 18 的 renderToPipeableStream
   - 服务端逐步发送HTML，浏览器逐步渲染
   - 不必等待所有数据加载完毕

2. 选择性水合（Selective Hydration）
   - React 18 只对交互部分进行水合
   - 非交互区域保持静态HTML

3. 渐进式水合（Progressive Hydration）
   - 按优先级逐步激活页面交互
   - 优先激活视口内组件

4. 岛屿架构（Islands Architecture）
   - 大部分页面为静态HTML
   - 只有交互组件需要水合
   - 代表框架：Astro、Qwik
```

### React 18 的流式 SSR

```jsx
// 服务端：流式传输HTML
import { renderToPipeableStream } from 'react-dom/server';

app.get('/', (req, res) => {
  const { pipe } = renderToPipeableStream(<App />, {
    bootstrapScripts: ['/main.js'],
    onShellReady() {
      // shell就绪后立即发送
      res.setHeader('content-type', 'text/html');
      pipe(res);
    },
    onShellError(error) {
      res.status(500).send('Error');
    },
  });
});

// Suspense边界可以让对应部分延迟流式传输
function App() {
  return (
    <div>
      <Header />  {/* 立即发送 */}
      <Suspense fallback={<Spinner />}>
        <SlowContent />  {/* 延迟发送 */}
      </Suspense>
      <Footer />  {/* Footer也需要等待？不！Streaming可以解决 */}
    </div>
  );
}
// 浏览器先看到Header，SlowContent显示Spinner
// SlowContent加载完成后，流式补充内容
```
