# ✅浏览器进程架构是怎样的？

# 典型回答

现代浏览器（以 Chrome 为例）采用**多进程架构**，每个进程负责不同的职责，通过进程间通信（IPC）进行协作。Chrome 的主要进程包括：

1. **浏览器进程（Browser Process）**：主进程，负责管理浏览器的界面窗口、标签页的创建和销毁、网络请求、文件访问、前进/后退等。浏览器进程中包含多个线程：
   - UI 线程：绘制地址栏、书签栏等界面
   - 网络线程：处理网络请求的收发
   - 存储线程：管理本地存储

2. **渲染进程（Renderer Process）**：每个标签页通常对应一个独立的渲染进程，负责 HTML 解析、CSS 样式计算、布局、绘制、JavaScript 执行等。渲染进程中包含多个线程：
   - 主线程（Main Thread）：处理 DOM 解析、JavaScript 执行、Layout、Paint 等
   - 合成线程（Compositor Thread）：负责图层合成，处理滚动和动画
   - 光栅线程（Raster Thread）：将指令转换为位图
   - Worker 线程：运行 Web Workers 或 Service Workers

3. **GPU 进程（GPU Process）**：负责处理 GPU 相关的任务，如 3D 渲染加速、视频解码等。

4. **网络进程（Network Process）**：在 Chrome 中，网络请求由浏览器进程中的网络线程管理（早期版本曾独立为网络进程）。

5. **插件进程（Plugin Process）**：每个插件（如 Flash）运行在独立进程中，防止插件崩溃影响主浏览器。

6. **扩展进程（Extension Process）**：每个浏览器扩展可以运行在独立的进程中。

# 扩展知识

### Chrome 多进程架构的优势

| 特性 | 说明 |
|------|------|
| 隔离性 | 单个标签页崩溃不会影响其他标签页或浏览器本身 |
| 安全性 | 渲染进程在沙箱中运行，无法直接访问系统资源 |
| 内存回收 | 关闭标签页时，其进程可以完全回收内存 |
| 性能 | 多个进程可以充分利用多核 CPU |
| 流畅性 | 合成线程独立于主线程工作，保证动画和滚动的流畅性 |

### 渲染进程内部的线程模型

```text
渲染进程 (Renderer Process)
│
├── 主线程 (Main Thread)
│   ├── HTML 解析器 → DOM 树
│   ├── CSS 解析器 → CSSOM 树
│   ├── JavaScript 引擎 (V8) → 执行 JS
│   ├── 样式计算 (Style)
│   ├── 布局 (Layout)
│   ├── 绘制记录 (Paint)
│   └── 事件处理 (Event Handler)
│
├── 合成线程 (Compositor Thread)
│   ├── 接收用户输入（滚轮、触摸）
│   ├── 动画处理（transform、opacity）
│   ├── 图层管理
│   └── 帧生成
│
├── 光栅线程池 (Raster Thread Pool)
│   └── 将绘制指令 → 位图 (Tile)
│
├── Worker 线程池
│   ├── Web Worker
│   ├── Service Worker
│   └── Worklet
│
└── 其他线程
    ├── 定时器线程 (setTimeout/setInterval)
    └── 网络请求线程 (XHR/Fetch)
```

### 各进程间的通信机制

```text
浏览器进程                   渲染进程
┌─────────────┐           ┌─────────────┐
│  UI 线程    │           │  主线程     │
│   │         │           │   │         │
│   ▼         │   IPC     │   ▼         │
│  网络线程 ──┼──────────►│  合成线程   │
│   │         │           │   │         │
│   ▼         │           │   ▼         │
│  存储线程   │           │  光栅线程   │
└─────────────┘           └─────────────┘
         │                       │
         │     GPU 进程          │
         │   ┌─────────────┐     │
         └──►│  GPU 线程   │◄────┘
             └─────────────┘
```

Chrome 使用 **Mojo** 作为 IPC 框架，通过命名管道（Unix Domain Socket 或 Windows Named Pipe）进行通信。

### 导航过程中的进程协作

```text
用户输入 URL -> 浏览器进程
  │
  1. 浏览器进程的 UI 线程：解析 URL，判断是搜索还是网址
  2. 浏览器进程的网络线程：发起网络请求，获取 HTML
  3. 浏览器进程：检查 URL 是否属于已存在的渲染进程（如同一站点）
  4. 分配或创建新的渲染进程
  5. 浏览器进程的 UI 线程：提交导航（Commit Navigation）
     │
     ▼
渲染进程开始工作
  6. 主线程：解析 HTML，构建 DOM 树
  7. 主线程：加载 CSS、执行 JavaScript
  8. 主线程：计算样式、布局、绘制
  9. 合成线程：合成图层
     │
     ▼
  10. IPC 通知浏览器进程：页面已渲染完成
  11. 浏览器进程更新标签页状态
```

### 站点隔离（Site Isolation）

从 Chrome 67 开始，**站点隔离**（Site Isolation）成为了默认功能。它使**同一标签页中不同站点的 iframe 也运行在独立的渲染进程中**：

```html
<!-- 主页面 example.com 的渲染进程 -->
<!-- 如果 iframe 来自不同源，Chrome 会为其分配独立的渲染进程 -->
<iframe src="https://different-origin.com"></iframe>
```

站点隔离的意义：
- 防御 **Spectre** 等侧信道攻击
- 每个 iframe 进程在沙箱中运行，无法访问其他进程的内存
- 不同站点的 Cookie、LocalStorage 被物理隔离

### 进程模型的资源开销

多进程架构的代价是更高的内存消耗：

| Chrome 进程 | 数量 | 典型内存占用 |
|-------------|------|-------------|
| 浏览器进程 | 1个 | 30-50MB |
| GPU 进程 | 1个 | 20-40MB |
| 网络进程 | 1个 | 5-10MB |
| 渲染进程 | 每个标签页1个 | 每个 20-80MB |
| 扩展进程 | 每扩展1个 | 每个 10-30MB |
| 插件进程 | 每个插件1个 | 每个 10-100MB |

Chrome 的优化策略：
- **进程复用**：同一站点（同一协议+域名+端口）的标签页可能共享渲染进程
- **进程丢弃**：后台标签页在内存不足时会被丢弃（Discard），再次激活时恢复
- **内存压缩**：使用 V8 的垃圾回收和压缩技术减少内存占用

### 与其他浏览器的对比

| 特性 | Chrome (Chromium) | Firefox | Safari |
|------|-------------------|---------|--------|
| 进程模型 | 多进程（站点隔离） | 多进程（Electrolysis） | 多进程 |
| 渲染进程隔离级别 | 每标签页+跨域iframe | 每标签页 | 每标签页 |
| GPU进程 | 独立 | 独立 | 浏览器进程内 |
| 网络进程 | 浏览器进程内 | 独立 | 浏览器进程内 |
| 插件进程 | 独立 | 独立 | NPAPI已废弃 |
| 沙箱 | 严格 | 有 | 有 |
