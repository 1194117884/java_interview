# ✅跨域解决方案有哪些？

# 典型回答

**跨域（Cross-Origin）** 是指浏览器出于安全考虑，实施的**同源策略（Same-Origin Policy）**限制。同源要求协议、域名、端口三者完全一致。当页面请求不同源的资源时，浏览器会阻止响应。

常见的跨域解决方案：

| 方案 | 原理 | 适用场景 | 是否需服务端配合 |
|------|------|---------|----------------|
| **CORS** | 服务端设置跨域响应头 | 主流方案，适用于 API 请求 | 是 |
| **JSONP** | 利用 `<script>` 标签不受同源限制 | 仅支持 GET 请求，兼容旧浏览器 | 是 |
| **反向代理** | 通过同源代理服务器转发请求 | 开发环境或生产环境 | 是（运维层面） |
| **postMessage** | 不同窗口/iframe 间通信 | 跨域窗口消息传递 | 否 |
| **WebSocket** | WebSocket 协议不受同源限制 | 需要实时通信的场景 | 是 |
| **document.domain** | 设置相同的 domain | 相同主域名不同子域名 | 否 |
| **location.hash / window.name** | 通过中转页面传递数据 | 旧浏览器兼容方案 | 否 |

最常用的是 **CORS**，它由服务端通过设置 HTTP 响应头来控制跨域访问权限，安全且功能完整。

# 扩展知识

### CORS 详解

CORS（Cross-Origin Resource Sharing，跨域资源共享）是 W3C 标准，通过在 HTTP 响应头中声明允许的来源来突破同源限制。

**简单请求与非简单请求：**

满足以下所有条件的为简单请求：
- 方法为 GET、HEAD 或 POST
- 除了自动携带的请求头外，只能设置 Accept、Accept-Language、Content-Language、Content-Type
- Content-Type 仅限于 application/x-www-form-urlencoded、multipart/form-data、text/plain

简单请求直接发送请求，浏览器检查响应头。非简单请求先发送 **预检请求（Preflight Request）**（OPTIONS 方法）。

```javascript
// 服务端 CORS 配置（Node.js Express）
app.use((req, res, next) => {
  // 允许的来源（生产环境应指定具体域名）
  const allowedOrigins = [
    'https://www.example.com',
    'https://admin.example.com'
  ];
  const origin = req.headers.origin;
  
  if (allowedOrigins.includes(origin)) {
    res.setHeader('Access-Control-Allow-Origin', origin);
  }
  
  // 允许携带凭证（Cookie、HTTP 认证）
  res.setHeader('Access-Control-Allow-Credentials', 'true');
  
  // 允许的 HTTP 方法
  res.setHeader('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE, OPTIONS');
  
  // 允许的自定义请求头
  res.setHeader('Access-Control-Allow-Headers', 'Content-Type, Authorization, X-Requested-With');
  
  // 预检请求的有效期（单位秒），减少 OPTIONS 请求次数
  res.setHeader('Access-Control-Max-Age', '86400');
  
  // 暴露给前端的响应头（默认前端只能访问 Cache-Control、Content-Type 等）
  res.setHeader('Access-Control-Expose-Headers', 'X-Total-Count, X-Request-Id');
  
  // 预检请求直接返回 204
  if (req.method === 'OPTIONS') {
    return res.status(204).end();
  }
  
  next();
});
```

```javascript
// 前端发送跨域请求（需要携带 Cookie）
fetch('https://api.example.com/data', {
  method: 'POST',
  credentials: 'include',  // 携带 Cookie
  headers: {
    'Content-Type': 'application/json',
    'X-Custom-Header': 'value',  // 自定义头会触发预检
  },
  body: JSON.stringify({ key: 'value' }),
})
.then(response => response.json())
.then(data => console.log(data))
.catch(error => console.error('跨域请求失败:', error));
```

### JSONP 的原理和局限

```javascript
// JSONP 的原理利用 <script> 标签不受同源限制

// 前端代码
function jsonp(url, params, callbackName) {
  return new Promise((resolve, reject) => {
    // 创建全局回调函数
    window[callbackName] = function(data) {
      resolve(data);
      // 清理
      delete window[callbackName];
      document.body.removeChild(script);
    };
    
    // 构建 URL
    const query = Object.entries(params)
      .map(([k, v]) => `${k}=${encodeURIComponent(v)}`)
      .join('&');
    
    const script = document.createElement('script');
    script.src = `${url}?${query}&callback=${callbackName}`;
    script.onerror = () => {
      reject(new Error('JSONP 请求失败'));
      delete window[callbackName];
      document.body.removeChild(script);
    };
    
    document.body.appendChild(script);
  });
}

// 使用
jsonp('https://api.example.com/users', { page: 1 }, 'handleUsers')
  .then(data => console.log(data));

// 服务端（Node.js）需要返回特定格式
// GET /users?page=1&callback=handleUsers
// 返回: handleUsers({ "users": [...], "total": 100 })
```

JSONP 的局限性：
- **仅支持 GET 请求**
- 存在安全风险（服务端不可控时可能执行恶意代码）
- 错误处理能力弱
- 不支持自定义请求头

### 反向代理解决方案

```javascript
// 开发环境：webpack-dev-server 代理配置
// webpack.config.js
module.exports = {
  devServer: {
    proxy: {
      '/api': {
        target: 'https://api.example.com',
        changeOrigin: true,       // 修改请求头中的 Host
        secure: true,            // 验证 SSL 证书
        pathRewrite: {
          '^/api': '',           // 重写路径
        },
        onProxyReq: (proxyReq, req, res) => {
          // 可以修改代理请求
          proxyReq.setHeader('X-Forwarded-For', req.ip);
        },
      },
    },
  },
};
```

```nginx
# 生产环境：Nginx 反向代理配置
server {
    listen 80;
    server_name www.example.com;
    
    # 前端静态资源
    location / {
        root /var/www/html;
        try_files $uri $uri/ /index.html;
    }
    
    # API 代理
    location /api/ {
        proxy_pass https://api.example.com/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        
        # 超时配置
        proxy_connect_timeout 60s;
        proxy_read_timeout 60s;
    }
}
```

### postMessage 跨域通信

```javascript
// 场景：父页面与嵌入的跨域 iframe 通信

// 父页面 (https://parent.com)
const iframe = document.getElementById('childFrame');
const childOrigin = 'https://child.com';

// 发送消息给 iframe
iframe.contentWindow.postMessage(
  { type: 'userInfo', data: { name: 'Alice', id: 123 } },
  childOrigin  // 目标源，'*' 表示不限制（不推荐）
);

// 接收来自 iframe 的消息
window.addEventListener('message', (event) => {
  // 务必验证来源
  if (event.origin !== childOrigin) return;
  
  console.log('收到子页面消息:', event.data);
}, false);
```

```javascript
// 子页面 (https://child.com)
const parentOrigin = 'https://parent.com';

// 接收来自父页面的消息
window.addEventListener('message', (event) => {
  // 验证来源
  if (event.origin !== parentOrigin) return;
  
  const { type, data } = event.data;
  
  if (type === 'userInfo') {
    // 处理数据...
    
    // 回复父页面
    event.source.postMessage(
      { type: 'ack', status: 'received' },
      event.origin
    );
  }
}, false);
```

### CORS 常见错误排查

| 错误信息 | 原因 | 解决方案 |
|---------|------|---------|
| No 'Access-Control-Allow-Origin' | 服务端未配置 CORS | 服务端添加 CORS 响应头 |
| Response to preflight doesn't pass | OPTIONS 请求未正确处理 | 服务端处理 OPTIONS 方法返回 204 |
| Credentials flag is 'true' | 携带凭证时 Origin 不能为 `*` | 明确指定具体域名 |
| Request header field X-Custom is not allowed | 使用了未在 Allow-Headers 中的请求头 | 在 Allow-Headers 中添加 |
| Method PATCH is not allowed | 请求方法不在 Allow-Methods 中 | 在 Allow-Methods 中添加 |
| Redirect is not allowed for a preflight | 预检请求遇到重定向 | 确保预检请求直接到达目标服务器 |

### 跨域与安全

```text
同源策略的本质是保护用户安全：
- 防止恶意网站读取其他网站的 Cookie（隐私泄露）
- 防止恶意网站操作其他网站的数据（CSRF）
- 防止恶意网站读取其他网站的 DOM（钓鱼攻击）

CORS 是在安全性和灵活性之间的平衡：
- 服务端通过白名单控制哪些来源可以访问
- 凭证（Cookie）仅在指定来源下发送
- 预检请求确认安全后才发送真实请求

最佳实践：
1. 不要使用 Access-Control-Allow-Origin: *
2. 明确指定允许的来源域名白名单
3. 只在需要时开启 credentials: include
4. 合理设置 Max-Age 减少预检请求
5. 敏感 API 结合 CSRF Token 使用
```
