# ✅浏览器渲染流水线是怎样的？

# 典型回答

浏览器的渲染流水线（Rendering Pipeline）是指从 HTML、CSS 和 JavaScript 等原始资源到最终屏幕上像素显示的完整处理流程。以 Chrome 的 Blink 渲染引擎为例，整个流水线分为以下几个核心阶段：

1. **DOM 树构建（Parsing）**：HTML 解析器将 HTML 字节流转换为 Token，再构建成 DOM（Document Object Model）节点树。
2. **CSSOM 树构建**：CSS 解析器将 CSS 字节流转换为 CSSOM（CSS Object Model），包含所有样式规则。
3. **渲染树构建（Attachment/Style）**：将 DOM 树与 CSSOM 树结合，计算出每个可见节点的最终样式（Computed Style）。
4. **布局（Layout）**：计算渲染树中每个节点的几何位置和尺寸，生成布局树（Layout Tree）。
5. **分层（Layer）**：根据层叠上下文和属性（如 `position: fixed`、`opacity`、`transform`）将节点分配到不同的图层。
6. **绘制（Paint）**：将每个图层的绘制指令记录下来（填充颜色、描边、绘制文本等）。
7. **合成（Composite）**：将各图层按照正确的层叠顺序合成为最终的位图，显示在屏幕上。

每个阶段都依赖前一阶段的结果，形成一条完整的流水线。

# 扩展知识

### 渲染流水线的触发方式

并非每次更新都会走完整个流水线。根据触发变更的属性不同，浏览器会走不同程度的流水线：

```text
JavaScript / Style / Layout / Paint / Composite

       |                            |
       v                            v
 只触发合成: transform, opacity  -----> Composite (最高性能)
       |                           
       v                           
 触发绘制+合成: color, background --> Paint -> Composite
       |
       v
 触发完整流水线: width, height, 
   margin, padding, left/top ------> Layout -> Paint -> Composite (最消耗性能)
```

### 构建DOM树的详细过程

DOM 树的构建是一个渐进式过程：

```javascript
// HTML字节流 -> Token -> Node -> DOM树
// 原始HTML: <html><body><p>Hello</p></body></html>

// 步骤1: 字节流解码 (Bytes -> Characters)
// <html><body><p>Hello</p></body></html>

// 步骤2: 词法分析 (Characters -> Tokens)
// StartTag(html) StartTag(body) StartTag(p) Text(Hello) EndTag(p) EndTag(body) EndTag(html)

// 步骤3: 语法分析 (Tokens -> Nodes)
// HTMLHtmlElement -> HTMLBodyElement -> HTMLParagraphElement -> Text("Hello")

// 步骤4: 节点组合 (Nodes -> DOM Tree)
// document
//   └── html
//         └── body
//               └── p
//                     └── "Hello"
```

DOM 树构建过程中遇到 `<script>` 标签会阻塞解析（除非标记为 `async` 或 `defer`），而 CSS 在默认情况下不会阻塞 DOM 解析，但会阻塞 JavaScript 的执行。

### CSSOM的构建和特性

CSSOM 构建会阻塞渲染树构建，因此 CSS 被认为是**渲染阻塞资源**：

```css
/* CSSOM树结构 */
/* 原始CSS */
body { font-size: 16px; }
.container { width: 100%; }
.container .title { color: red; }

/* CSSOM树（简化表示） */
/* Rule树会级联继承，每个节点最终计算得到完整的样式集 */
```

CSS 解析特点：
- CSS 选择器从**右向左**匹配（从最具体的部分开始，减少回溯）
- CSSOM 的构建是同步的，会阻塞渲染
- 媒体查询中的 CSS 可以不阻塞渲染（如 `<link rel="stylesheet" media="print">`）

### 渲染树（Render Tree）与DOM树的区别

| 特性 | DOM树 | 渲染树 |
|------|-------|--------|
| 节点类型 | DOM节点 | RenderObject |
| 包含范围 | 所有节点 | 仅可见节点 |
| 包含样式信息 | 否 | 是 |
| 隐藏元素 | 存在 | 不存在（`display:none` 被剔除） |
| 伪元素 | 不存在 | 存在（`::before`、`::after` 会出现在渲染树中） |
| `visibility:hidden` 元素 | 存在 | 存在（隐藏但占位） |

### 图层创建的条件

浏览器会将以下情形的元素创建为独立的合成层（Compositing Layer）：

```css
/* 触发合成层创建的常见条件 */
.element1 { transform: translateZ(0); }      /* 3D变换 */
.element2 { will-change: transform; }        /* will-change提示 */
.element3 { position: fixed; }               /* 固定定位 */
.element4 { opacity: 0.8; }                 /* 非1的透明度 */
.element5 { /* 有滚动条的元素 */ overflow: scroll; }
.element6 { filter: blur(5px); }            /* CSS滤镜 */
.element7 { /* <video>或<canvas>元素 */ }
```

合理使用合成层可以提升性能：

```css
/* 推荐的做法：用transform代替top/left做动画 */
/* ❌ 不推荐 - 触发Layout */
.box { animation: slide 1s; }
@keyframes slide {
  from { left: 0; }
  to   { left: 100px; }
}

/* ✅ 推荐 - 仅触发Composite */
.box { animation: slide 1s; }
@keyframes slide {
  from { transform: translateX(0); }
  to   { transform: translateX(100px); }
}
```

### 合成线程与主线程的协作

现代浏览器的渲染是多线程协作的结果：

| 线程 | 职责 | 可访问资源 |
|------|------|------------|
| 主线程（Main Thread） | DOM解析、CSS计算、Layout、Paint记录、JavaScript执行 | DOM、CSSOM |
| 合成线程（Compositor Thread） | 图层合成、滚动处理、动画执行 | 图层位图 |
| 光栅线程（Raster Thread） | 将绘制指令转换为位图 | 绘制指令 |

合成线程的优势在于它可以独立于主线程工作。当用户滚动页面或执行 `transform`、`opacity` 动画时，合成线程可以直接操作已有位图，无需主线程参与，保证了动画的流畅性（60fps）。
