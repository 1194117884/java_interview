# ✅关键渲染路径优化策略

# 典型回答

关键渲染路径（Critical Rendering Path, CRP）是指浏览器从 HTML、CSS、JavaScript 等资源开始，到首次渲染出像素所经历的一系列步骤。优化 CRP 的核心目标是**尽可能缩短首次渲染的时间**（即 First Paint 或 First Contentful Paint）。

关键渲染路径优化的主要策略包括：

1. **减少关键资源数量**：内联关键 CSS 和 JS，延迟非关键资源加载
2. **减少关键字节数**：压缩、混淆、Tree Shaking、移除无用代码
3. **缩短关键路径长度**：异步加载非关键脚本、合并小文件减少请求数
4. **优化资源加载顺序**：确保关键资源优先加载，CSS 在 JS 之前加载

# 扩展知识

### 关键渲染路径的衡量指标

```javascript
// 使用 Performance API 评估 CRP
const perf = performance.getEntriesByType('navigation')[0];
const crpMetrics = {
  // 关键资源总耗时
  total: perf.loadEventEnd - perf.navigationStart,
  // 首字节时间（TTFB）
  ttfb: perf.responseStart - perf.requestStart,
  // 首屏渲染时间
  firstPaint: perf.domInteractive - perf.navigationStart,
  // 关键资源加载完成时间
  criticalComplete: perf.domContentLoadedEventEnd - perf.navigationStart,
};
console.table(crpMetrics);
```

### 优化CSS传递

CSS 是渲染阻塞资源，优化 CSS 是 CRP 优化的关键：

```html
<!-- ❌ 不推荐：外部CSS需要额外网络请求 -->
<link rel="stylesheet" href="styles.css">

<!-- ✅ 推荐：关键CSS内联到 head 中 -->
<head>
  <style>
    /* 首屏关键CSS直接内联 */
    .header { position: fixed; top: 0; width: 100%; }
    .hero { height: 100vh; display: flex; align-items: center; }
    /* 仅包含首屏可见区域的样式 */
  </style>
  <!-- 非关键CSS异步加载 -->
  <link rel="preload" href="non-critical.css" as="style" 
        onload="this.onload=null;this.rel='stylesheet'">
  <noscript><link rel="stylesheet" href="non-critical.css"></noscript>
</head>
```

### 优化 JavaScript

```html
<!-- ✅ 使用 defer 加载需要DOM的脚本 -->
<script defer src="app.js"></script>

<!-- ✅ 使用 async 加载独立脚本 -->
<script async src="analytics.js"></script>

<!-- ✅ 内联关键 JavaScript -->
<script>
  // 必须立即执行的逻辑（如判断是否移动端）
  if (window.innerWidth < 768) {
    document.documentElement.classList.add('is-mobile');
  }
</script>

<!-- ❌ 不推荐：同步脚本放在 head 中 -->
<script src="large-library.js"></script>
```

### 完整的 CRP 优化清单

| 优化项 | 说明 | 效果 |
|--------|------|------|
| 内联关键CSS | 将首屏CSS放入 `<style>` 标签 | 减少1次RTT |
| 异步非关键CSS | 使用 `preload` + `onload` 加载非首屏CSS | 不阻塞渲染 |
| 延迟JS执行 | 使用 `defer`/`async` 属性 | 不阻塞HTML解析 |
| 内联关键JS | 将关键的JS代码直接内联 | 减少额外请求 |
| 资源预加载 | 使用 `<link rel="preload">` 预加载关键资源 | 提前网络请求 |
| 预连接 | 使用 `<link rel="preconnect">` 提前建立连接 | 减少DNS/TCP/TLS耗时 |
| 压缩 | 使用 Gzip/Brotli 压缩文本资源 | 减少传输量 60-80% |
| Tree Shaking | 移除未使用的代码 | 减少JS体积 |
| 代码分割 | 按路由拆分 JS chunk | 减少首屏JS大小 |
| HTTP/2 服务器推送 | 主动推送关键资源 | 减少请求轮次 |

### 使用媒体查询解除CSS阻塞

```html
<!-- 如果样式仅在某些条件下使用，添加 media 属性 -->
<!-- 打印样式不会阻塞屏幕渲染 -->
<link rel="stylesheet" href="print.css" media="print">

<!-- 响应式CSS，仅在宽度匹配时阻塞渲染 -->
<link rel="stylesheet" href="desktop.css" media="(min-width: 768px)">
<link rel="stylesheet" href="mobile.css" media="(max-width: 767px)">

<!-- orientation 查询 -->
<link rel="stylesheet" href="landscape.css" media="(orientation: landscape)">
```

### 提取关键CSS的工具化方案

```javascript
// 使用 Critical 工具（Node.js 构建时）
// npm install -D critical

const critical = require('critical').stream;

gulp.task('critical', () => {
  return gulp.src('dist/**/*.html')
    .pipe(critical({
      base: 'dist/',
      inline: true,        // 自动内联关键CSS
      css: ['dist/styles.css'],
      width: 375,           // 移动端视口宽度
      height: 812,          // 移动端视口高度
      minify: true,
    }))
    .pipe(gulp.dest('dist'));
});
```

### 资源提示（Resource Hints）详解

```html
<!-- dns-prefetch：提前进行DNS解析 -->
<link rel="dns-prefetch" href="//fonts.googleapis.com">

<!-- preconnect：提前建立连接（DNS + TCP + TLS） -->
<link rel="preconnect" href="//api.example.com">

<!-- preload：提前加载当前页面需要的资源 -->
<link rel="preload" href="hero.jpg" as="image">
<link rel="preload" href="fonts/roboto.woff2" as="font" crossorigin>
<link rel="preload" href="critical.css" as="style">

<!-- prefetch：提前获取未来导航可能需要的资源（低优先级） -->
<link rel="prefetch" href="next-page.html" as="document">
<link rel="prefetch" href="next-page-chunk.js" as="script">

<!-- prerender：预渲染整个页面（极高的资源消耗，谨慎使用） -->
<link rel="prerender" href="likely-next-page.html">
```

### 首帧渲染的时间线对比

```text
优化前的时间线：
|-- DNS --|-- TCP --|-- TLS --|-- 请求HTML --|-- 解析HTML(遇到CSS) --|-- 请求CSS --|-- CSS解析 --|-- 请求JS --|-- JS执行 --|-- 首次渲染 --|
                    ↑ 关键资源串行加载，总耗时 = 多个RTT之和

优化后的时间线：
|-- DNS --|-- TCP --|-- TLS --|-- 请求HTML --|-- 解析HTML(同时preconnect) --|-- 内联CSS --|-- JS(defer) --|-- 首次渲染 --|
    ↑ preconnect           ↑ preload                         ↑ 关键资源在1个RTT内准备就绪
```

### 实际案例：首屏优化实践

```html
<!-- 优化后的 HTML 结构 -->
<!DOCTYPE html>
<html>
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  
  <!-- 1. 预连接到第三方源 -->
  <link rel="preconnect" href="https://fonts.googleapis.com">
  <link rel="dns-prefetch" href="https://analytics.example.com">
  
  <!-- 2. 内联关键CSS -->
  <style>
    /* 首屏可见区域的关键样式 */
    /* ... 约 2-4KB 的关键CSS */
  </style>
  
  <!-- 3. 异步非关键CSS -->
  <link rel="preload" href="full.css" as="style" 
        onload="this.onload=null;this.rel='stylesheet'">
  <noscript><link rel="stylesheet" href="full.css"></noscript>
  
  <!-- 4. 首屏必需的JS（非常少量） -->
  <script>
    // 最小化首屏必需JS，通常仅处理字体闪白或critical交互
  </script>
</head>
<body>
  <!-- 5. 首屏HTML内容 -->
  <header>...</header>
  <main>...</main>
  
  <!-- 6. 非关键资源放在底部 -->
  <script defer src="app.js"></script>
</body>
</html>
```
