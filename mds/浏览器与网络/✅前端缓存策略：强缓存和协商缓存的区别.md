# ✅前端缓存策略：强缓存和协商缓存的区别

# 典型回答

浏览器缓存是前端性能优化的重要手段，分为**强缓存**和**协商缓存**两种机制。两者的核心区别在于是否需要向服务器发送请求来验证缓存是否可用。

**强缓存（Strong Caching）**
- 工作原理：浏览器根据响应头中的 `Expires` 或 `Cache-Control: max-age` 判断缓存是否有效
- 请求过程：缓存有效期内**不会向服务器发送请求**，直接从本地缓存读取
- 状态码：200（from disk cache / from memory cache）
- 优点：零网络延迟，加载最快
- 缺点：缓存过期前无法获取服务器上的最新资源

**协商缓存（Negotiation Caching）**
- 工作原理：浏览器向服务器发送请求，由服务器判断缓存是否可用
- 请求过程：**每次都会发送请求**（但响应体很小），服务器返回 `304 Not Modified` 时使用缓存
- 状态码：304 Not Modified
- 优点：可以获取最新的资源
- 缺点：每次都需要网络请求（至少发送 HTTP 头），相对强缓存更慢

# 扩展知识

### 强缓存相关响应头

```javascript
// Expires（HTTP/1.0，已逐渐被 Cache-Control 取代）
// 绝对时间，依赖于服务器时间
Expires: Wed, 21 Oct 2025 07:28:00 GMT

// Cache-Control（HTTP/1.1，优先级高于 Expires）
// 相对时间
Cache-Control: max-age=3600          // 缓存 1 小时
Cache-Control: max-age=86400         // 缓存 1 天
Cache-Control: max-age=31536000      // 缓存 1 年（适用于带哈希的资源）

// Cache-Control 其他指令
Cache-Control: no-cache              // 不使用强缓存，使用协商缓存
Cache-Control: no-store              // 完全不缓存（敏感数据）
Cache-Control: public                // 任何中间节点（CDN、代理）都可以缓存
Cache-Control: private               // 仅浏览器可以缓存，中间节点不可缓存
Cache-Control: must-revalidate       // 缓存过期后必须向源服务器验证
Cache-Control: immutable             // 内容永不改变（配合 max-age 使用）
```

### 协商缓存相关响应头

```javascript
// 方案1: Last-Modified / If-Modified-Since（基于时间）
// 第一次响应：
Last-Modified: Wed, 21 Oct 2024 07:28:00 GMT

// 后续请求：
If-Modified-Since: Wed, 21 Oct 2024 07:28:00 GMT

// 服务器响应：
304 Not Modified  // 文件未修改，使用缓存
200 OK            // 文件已修改，返回新内容

// 方案2: ETag / If-None-Match（基于内容哈希，推荐）
// 第一次响应：
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"

// 后续请求：
If-None-Match: "33a64df551425fcc55e4d42a148795d9f25f89d4"

// 服务器响应：
304 Not Modified  // ETag 匹配，使用缓存
200 OK            // ETag 不匹配，返回新内容
```

### 缓存机制的优先级

```text
浏览器请求资源时的缓存判断流程：

1. 是否有 Cache-Control / Expires？
   ├── 有且未过期 → 强缓存生效 ✅（200 from cache）
   └── 无或已过期
        │
        └── 2. 是否有 ETag / Last-Modified？
             ├── 有 → 发送协商请求
             │    ├── 服务器判定未修改 → 304 Not Modified ✅
             │    └── 服务器判定已修改 → 200 OK（返回新资源）
             └── 无 → 直接请求服务器
```

### 不同资源的缓存策略配置

```javascript
// Nginx 缓存配置示例
server {
  // HTML：不缓存或短缓存（内容可能频繁变化）
  location / {
    add_header Cache-Control "no-cache, must-revalidate";
  }
  
  // 图片：长时间缓存
  location ~* \.(jpg|jpeg|png|gif|ico|svg)$ {
    expires 30d;
    add_header Cache-Control "public, immutable";
  }
  
  // 字体：长时间缓存
  location ~* \.(woff|woff2|ttf|eot)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
    add_header Access-Control-Allow-Origin "*";
  }
  
  // CSS/JS（带哈希）：永久缓存
  location ~* \.(css|js)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
  }
}
```

### 构建工具中的缓存策略

```javascript
// webpack 配置：通过 content hash 实现永久缓存
module.exports = {
  output: {
    filename: 'js/[name].[contenthash:8].js',
    chunkFilename: 'js/[name].[contenthash:8].chunk.js',
  },
  plugins: [
    // 提取 CSS 到单独文件
    new MiniCssExtractPlugin({
      filename: 'css/[name].[contenthash:8].css',
    }),
  ],
};

// webpack 打包结果：
// js/main.a1b2c3d4.js
// js/vendor.e5f6g7h8.js
// css/main.i9j0k1l2.css

// 资源内容变化 → contenthash 变化 → 全新的 URL
// 资源内容不变 → contenthash 不变 → 强缓存继续有效
```

### Service Worker 缓存策略

```javascript
// Service Worker 中的缓存策略

// 1. Cache First（缓存优先）
// 适用于：不变的静态资源、图片
async function cacheFirst(request) {
  const cached = await caches.match(request);
  if (cached) return cached;
  
  const response = await fetch(request);
  const cache = await caches.open('static-v1');
  cache.put(request, response.clone());
  return response;
}

// 2. Network First（网络优先）
// 适用于：经常变化但需保证可靠性的资源
async function networkFirst(request) {
  try {
    const response = await fetch(request);
    const cache = await caches.open('dynamic-v1');
    cache.put(request, response.clone());
    return response;
  } catch (error) {
    const cached = await caches.match(request);
    if (cached) return cached;
    throw error;
  }
}

// 3. Stale While Revalidate（更新缓存的同时返回旧缓存）
// 适用于：不要求即时更新的资源（用户头像、文章列表）
async function staleWhileRevalidate(request) {
  const cache = await caches.open('runtime-v1');
  
  // 立即返回缓存（如果有）
  const cachedResponse = await cache.match(request);
  const fetchPromise = fetch(request).then(response => {
    cache.put(request, response.clone());
    return response;
  });
  
  return cachedResponse || fetchPromise;
}

// 4. Network Only（仅网络）
// 适用于：API 请求、需要实时数据
function networkOnly(request) {
  return fetch(request);
}

// 5. Cache Only（仅缓存）
// 适用于：预缓存的静态资源
async function cacheOnly(request) {
  return await caches.match(request);
}
```

### 缓存策略选择矩阵

```text
资源类型    缓存策略                  原因
─────────────────────────────────────────────────────
HTML        no-cache 或 max-age=0    内容可随时更新
JSON API    no-cache                 保证数据实时性
带哈希的JS   max-age=1y immutable    文件名=版本号，永不失效
带哈希的CSS  max-age=1y immutable    同上
图片(头像)  max-age=7d stale-while-revalidate  可接受短时延迟
图片(产品)  max-age=30d immutable    更新不频繁
字体        max-age=1y immutable    几乎不变化
用户数据     no-store                 禁止缓存
```

### 如何验证缓存的命中情况

```javascript
// 浏览器中通过控制台查看缓存状态

// 方法1: 使用 Performance API
const entries = performance.getEntriesByType('resource');
entries.forEach(entry => {
  // 判断是否从缓存加载
  const isDiskCache = entry.transferSize === 0 && entry.duration > 0;
  const isMemoryCache = entry.transferSize === 0 && entry.duration === 0;
  const is304 = entry.transferSize > 0; // 发生了网络传输
  
  console.log(`${entry.name}:`, {
    transferSize: entry.transferSize,  // 传输字节数（0 = 缓存）
    duration: entry.duration,          // 加载耗时
    cacheType: isDiskCache ? 'disk cache' : 
               isMemoryCache ? 'memory cache' : 'network/304',
  });
});

// 方法2: 在 Chrome DevTools 中查看
// Network 面板 → Size 列
// "disk cache" = 磁盘缓存
// "memory cache" = 内存缓存
// "(from service worker)" = 来自 Service Worker
// 数字（如 "12.3 KB"）= 来自网络
// "304" = 协商缓存命中
```

### 缓存相关的常见问题

```text
问题1: 用户看不到最新版本
原因: 强缓存未过期，浏览器使用旧缓存
解决:
  - 使用带 contenthash 的文件名
  - 缩短 max-age 时间
  - 在 HTML 中添加版本号参数（不够可靠）
  - 使用 Service Worker 主动更新

问题2: 强缓存和协商缓存混用导致问题
原因: 对同一资源同时设置了长 max-age 和 ETag
解决: 根据资源更新频率选择合适的策略

问题3: CDN 缓存与浏览器缓存冲突
原因: CDN 缓存未清空导致用户获取到旧资源
解决:
  - 使用版本化 URL
  - 配置 CDN 的缓存规则
  - 使用 CDN 的刷新/预热功能
```
